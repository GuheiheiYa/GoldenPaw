# 常见陷阱和最佳实践

GoldenPaw 项目中常见的错误和推荐做法。

---

## 必须避免的错误

### 1. 未使用技能直接开始

**错误**: 直接开始编码，不调用相关技能

**正确做法**:
- 前端 UI 开发 → 先调用 `/frontend-design`
- 新功能开发 → 先调用 `/feature-dev` 或 `/writing-plans`
- 遇到 bug → 先调用 `/systematic-debugging`

### 2. 跳过测试直接提交

**错误**: 写完代码就提交，不验证

**正确做法**:
- 实现完成后 → 调用 `/code-review`
- 提交前 → 调用 `/verification-before-completion`
- 运行测试 → `npx vitest run`

### 3. 一次修改多个功能

**错误**: 一个提交包含多个不相关的功能

**正确做法**:
- 每次只做一个功能
- 每个功能单独提交
- 便于回退和追踪

### 4. 在主分支开发

**错误**: 直接在 master 分支开发

**正确做法**:
- 使用 feature 分支
- 或使用 worktree（`/using-git-worktrees`）
- 完成后合并到 master

### 5. 不更新文档

**错误**: 只更新代码，不更新文档

**正确做法**:
- 每次代码变更都必须同步文档
- 见 `docs/rules/docs-workflow.md`
- 提交前检查文档

### 6. 使用文字描述代替 ID

**错误**: 在文档中只用文字描述需求/问题

**正确做法**:
- 必须使用 `[R-xxx]`、`[F-xxx]`、`[ISS-xxx]`
- 便于追踪和关联
- 见 `docs/rules/docs-workflow.md`

### 7. 使用 `git add -A` 或 `git add .`

**错误**: 暂存所有文件，可能包含敏感文件

**正确做法**:
- 使用 `git add <specific-file>`
- 只暂存需要提交的文件
- 检查 `git status` 确认

### 8. 使用 `git push --force`

**错误**: 强制推送，覆盖远程历史

**正确做法**:
- 使用 `git revert` 创建新提交来撤销
- 避免强制推送
- 如果必须，使用 `git push --force-with-lease`

## 最佳实践

### 1. 先计划后实现

**原则**: 开始前先制定计划

**做法**:
- 使用 `/writing-plans` 制定实现计划
- 定义成功标准
- 列出需要的文件和改动

### 2. 边做边测

**原则**: 开发过程中持续测试

**做法**:
- 使用 `/test-driven-development`（可测试的逻辑）
- 每次改动后运行相关测试
- 不要等完成后再测试

### 3. 频繁提交

**原则**: 小步提交，便于追踪和回退

**做法**:
- 每个功能或修复单独提交
- 提交信息清晰说明做了什么
- 便于回退和 bisect

### 4. 代码审查前先自查

**原则**: 提交前先自己审查一遍

**做法**:
- 使用 `/code-review` 审查代码
- 检查代码质量、性能、安全
- 修复发现的问题

### 5. 验证后再标记完成

**原则**: 确保功能正常后再标记完成

**做法**:
- 使用 `/verification-before-completion`
- 运行相关测试
- 手动验证关键流程

### 6. 文档同步更新

**原则**: 代码和文档同步更新

**做法**:
- 每次代码变更都更新文档
- 见 `docs/rules/docs-workflow.md`
- 提交前检查文档是否已更新

## 性能优化建议

### 1. 避免不必要的重渲染

**错误**: 在 Vue 组件中使用响应式对象时频繁创建新对象

**正确做法**:
- 使用 `shallowRef` 或 `markRaw` 避免深度响应式
- 使用 `computed` 缓存计算结果
- 避免在模板中使用方法调用

### 2. 使用懒加载

**错误**: 一次性加载所有组件和数据

**正确做法**:
- 使用 `defineAsyncComponent` 异步加载组件
- 使用路由懒加载
- 分页加载列表数据

### 3. 优化图片和资源

**错误**: 使用未压缩的图片

**正确做法**:
- 使用 WebP 格式
- 压缩图片
- 使用 CDN

## 安全建议

### 1. 不要硬编码敏感信息

**错误**: 在代码中硬编码 API 密钥、密码

**正确做法**:
- 使用环境变量
- 使用 `.env` 文件（不要提交到 git）
- 使用 `import.meta.env.VITE_XXX`

### 2. 验证用户输入

**错误**: 直接使用用户输入，不验证

**正确做法**:
- 验证所有用户输入
- 使用类型检查
- 防止 XSS 和注入攻击

### 3. 使用 HTTPS

**错误**: 使用 HTTP 传输敏感数据

**正确做法**:
- 所有 API 调用使用 HTTPS
- 不要在 HTTP 中传输密码、令牌

---

## 调试建议

### 1. 使用系统化调试

**错误**: 随意尝试修复，不分析根本原因

**正确做法**:
- 使用 `/systematic-debugging`
- 定位根本原因
- 编写测试防止回归

### 2. 使用 console.log 调试

**错误**: 遗留 console.log 在生产代码中

**正确做法**:
- 调试时使用 console.log
- 提交前删除所有 console.log
- 使用日志库（如需要）

### 3. 使用断点调试

**错误**: 只使用 console.log

**正确做法**:
- 使用 IDE 断点调试
- 更高效定位问题
- 可以查看变量状态

## 检查清单

### 开发前

- [ ] 调用相关技能（`/frontend-design`、`/feature-dev`、`/writing-plans`）
- [ ] 制定实现计划
- [ ] 确定是否需要 TDD
- [ ] 在 feature 分支或 worktree 开发

### 开发中

- [ ] 每次改动后运行相关测试
- [ ] 边做边更新文档
- [ ] 避免一次修改多个功能
- [ ] 使用有意义的变量和函数名

### 完成后

- [ ] 调用 `/code-review` 审查代码
- [ ] 调用 `/verification-before-completion` 验证
- [ ] 运行所有相关测试
- [ ] 更新所有相关文档

### 提交前

- [ ] 检查 `git status` 和 `git diff`
- [ ] 使用 `git add <specific-file>` 暂存
- [ ] 编写清晰的提交信息
- [ ] 更新 `git-commits.md`
- [ ] 推送到远程
